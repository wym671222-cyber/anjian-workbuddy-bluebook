# 第 22 章 Vibe Coding：从想法到上线

这一章只解决一个问题：**不会写代码，怎样用 WorkBuddy 把一个想法变成一个能打开、能访问、能持续迭代的产品。**

不需要先学编程。跟着下面两个合成示例操作一次，就能掌握"一句话需求 → 规划确认 → 本地验收 → 部署上线 → 分支迭代"的完整链路。

本章所有产品名、域名、IP 和数字均为合成示例，提示词均以可复制文本框呈现。Vibe Coding 指用自然语言描述需求、由 AI 完成编码的工作方式，你是产品经理，AI 是执行者。

::: warning 先记住一条
让 AI 写代码不等于放弃判断。每一个版本生成前先检查计划，等我确认后再执行；生成的文件不覆盖已有版本；看不懂的技术决策标记"待确认"，问清楚再动手。涉及域名、服务器、备案的操作花的是真钱、走公网，必须人工确认每一步。
:::

## 30 秒看懂本章

| 你遇到的问题 | 跟做哪个示例 | 最后会得到什么 |
|---|---|---|
| 有个小想法，想先做个能跑的页面 | 示例一 | 一个本地可预览的单页应用和私有代码仓库 |
| 产品要给同事用，得放到网上 | 示例二 | 部署到服务器、绑定域名和 HTTPS 的可访问版本 |

第一次练习只要七步：

```mermaid
flowchart LR
    A[新建项目文件夹] --> B[一句话描述需求]
    B --> C[确认规划文档]
    C --> D[生成并本地验收]
    D --> E[推送到私有仓库]
    E --> F[部署到服务器]
    F --> G[分支迭代循环]
```

## 开始前只做四件事

1. 新建一个文件夹，用产品名命名，例如 `zhouji-notes`。这个文件夹就是产品的家，代码、配置、文档都在这里。
2. 打开 WorkBuddy，把这个文件夹设为工作目录（操作见第 3 章）。
3. 想清楚一句话需求：给谁用、解决什么问题、最核心的一个功能。第一版故意简单，功能后面慢慢加。
4. 如果要正式上线，提前准备：一个云厂商账号（买域名和服务器）、一个代码托管账号（如 GitHub，建**私有**仓库）。备案、价格与平台规则以各官方页面为准。

::: tip 文本框怎么用
下面深色文本框右上角有"复制"按钮。点击后粘贴到 WorkBuddy；产品名、域名与实际情况不同时，只改对应名称，不要删掉检查和验收要求。
:::

## 示例一：两小时做出"周记便签"单页应用

合成场景：做一个给自己用的周记网页，能按周记录、查看历史，先在自己电脑上跑起来。

### 1. 先规划，不直接写代码

用 WorkBuddy 的规划模式（Plan 模式），用大白话描述需求，让它先产出计划而不是直接动手。

**复制提示词 1：先规划后执行**

```text
我想做一个叫"周记便签"的单页网页应用，给自己用。

需求：按周记录工作和生活事项，能新增、编辑、删除，能按周查看历史；数据保存在浏览器本地，不需要登录。

请先不要写代码。只输出一份规划文档 plan.md，包括：
1. 页面结构和每个区域的作用；
2. 数据存在哪里、有哪些字段；
3. 你打算用的技术方案和理由（用外行能懂的话解释）；
4. 实施步骤清单。

我看过后确认，你再开始执行。
```

WorkBuddy 生成 plan.md 后，逐条检查：页面结构是不是你想要的？有没有自作主张加功能？看不懂的术语直接问。确认方向对了再发第二段。

### 2. 确认后执行，本地预览验收

**复制提示词 2：按确认的规划执行**

```text
plan.md 我看过了，确认按这个方向执行。

请开始开发，完成后：
1. 启动本地预览，告诉我访问地址；
2. 在 README.md 里写清这个项目是什么、怎么启动、文件结构；
3. 报告生成了哪些文件、每个文件的作用。
```

在浏览器打开预览地址（通常是 `localhost:3000` 之类的地址），以产品经理身份逐项验收：新增一条周记、刷新页面看数据还在不在、删掉一条看列表对不对。不满意就直说，"UI 太挤了""删除按钮不明显"，让它改到你满意为止。

::: tip 验收只看三点
1. 页面的每个功能你都亲手点过一遍；2. 刷新后数据表现符合预期；3. README 里的启动方法你照着能复现。
:::

### 3. 代码不能只留在自己电脑上

**复制提示词 3：推送到私有仓库**

```text
请初始化 Git 仓库，连接我的 GitHub 账号，创建一个私有仓库（注意是私有，不是公开），把当前代码提交为第一个版本并推送上去。

提交前告诉我 .gitignore 里排除了哪些文件，确认没有把账号、密钥类内容提交进去。
```

## 示例二：把"物品借用登记"部署上线

合成场景：给小组做一个物品借用登记网页，同事通过浏览器访问，需要服务器、域名和 HTTPS。这个示例包含真实上线必经的行政步骤和外发控制。

### 1. 先定名字，再查重

名字会绑定域名、备案和后续一切，改名成本极高。先让 WorkBuddy 帮你检查候选名字：

**复制提示词 4：起名查重（只检查不注册）**

```text
我准备给物品借用登记工具起名，候选："易借通""班组百宝箱""借还小站"。

请帮我逐一检查并列出检查结论：
1. 常见应用商店、公众号、小程序里有没有同名或高度相似；
2. 对应的常见后缀域名是否已被注册、是否在合理预算内；
3. 给出你的推荐和理由。

只检查，不要替我注册任何东西。
```

确定名字后，人去云厂商控制台注册域名（本例使用合成域名 `yijie.example.cn` 演示）。如果产品面向中国大陆用户提供服务，域名需要完成 ICP 备案才能正常解析访问；在买服务器的云平台找到备案入口按指引提交，审核需要等待，**务必与开发同步进行**，别等产品做完才想起来。出海产品则选择离目标用户近的海外节点，不涉及的环节跳过。

### 2. 开发几轮之后，先整理文档再继续

跟 AI 聊了很多轮、改了很多东西之后，代码和文档大概率已经乱了，甚至互相矛盾。AI 的记忆有限，新开对话时之前的设计决策它会忘记，没有干净的文档兜底，它就会写出和之前矛盾的代码。

先做一次规范化整理再继续：

**复制提示词 5：整理项目文档**

```text
请先只读检查整个项目，不要改代码。

然后帮我整理项目文档：
1. 更新 README.md：项目现状、已完成的功能、运行方式；
2. 把分散在对话里的设计决策整理成 docs/decisions.md，并删除互相矛盾的旧表述；
3. 列出文档与代码不一致的地方，标记"待确认"，等我逐条回复。

整理完报告改动清单，不要覆盖原文件，生成新版本。
```

这类"整理文档"的动作每个项目都要反复做，值得按第 23 章的方法做成自己的 Skill；也可以参考公开来源中卡兹克开源的 neat-freak Skill（见文末参考资料）的思路。每次新对话开始，AI 读一遍文档就能恢复上下文。

### 3. 上线前，补上测试和分支保护

**复制提示词 6：搭建测试与分支保护**

```text
请为这个项目做两件事：

1. 搭一套测试流程：为核心功能写基础测试，配置 CI，让每次提交自动跑测试；
2. 帮我在仓库设置里开启主分支保护：任何人（包括我）都不能直接推送到 main，所有改动必须走 PR，PR 通过 CI 测试后才允许合并。

完成后告诉我：测试怎么跑、分支保护规则是什么、以后每次改功能的正确流程。
不懂的名词（分支、PR、CI）请用外行能懂的话解释给我听。
```

没有分支保护时，你或 AI 随时可能手一抖把有问题的代码推上主分支，线上直接崩掉。有了分支保护，这条路被堵死：改坏了就扔掉分支重来，主分支永远是稳定能跑的版本。

### 4. 部署、绑域名、配 HTTPS（每一步都确认）

把服务器 IP（本例使用文档专用合成地址 `203.0.113.10`）和域名发给 WorkBuddy，让它处理部署，但每一步都要它先报告再执行：

**复制提示词 7：部署上线**

```text
我要把项目部署到服务器 203.0.113.10，并把域名 yijie.example.cn 指向它，配好 HTTPS 免费证书。

要求：
1. 每一步操作前先说明要做什么、改什么配置，等我确认后再执行；
2. 需要登录云控制台的操作列成清单，由我自己完成；
3. 部署完成后核对：服务是否 7x24 运行、域名是否解析成功、HTTPS 是否生效，逐项报告结果；
4. 把部署步骤写进 docs/deploy.md，以后照这份文档能重新部署。
```

在浏览器输入自己的域名，看到产品真的打开——这是 Vibe Coding 最爽的时刻。

### 5. 进入迭代循环

上线不是终点，是起点。之后每一次改版、每一个新功能、每一个 bug 修复，都走同一条循环：

```text
新建分支 → 说出需求 → 开发完成 → 跑一遍文档整理 → 提交 PR 过 CI 测试 → 合并到主分支 → 部署上线
```

这个流程最大的好处是安全：改动都在分支上，主分支永远稳定；文档每轮整理一次，AI 永远不会"忘了之前的设计"。跑两三遍就会形成肌肉记忆，还能稳定地同时推进多个任务。

## 可以不懂代码，但必须懂架构

这是本章最重要的一段。你可以让 AI 写所有的代码，但你必须知道产品是怎么运转的：每个页面对应哪个功能，数据库有哪些表、存了哪些字段，用户点了一个按钮之后数据怎么流转。

如果你不清楚，就丧失了对产品的判断力——AI 说"改好了"，你连验收都验收不了；AI 提一个技术方案，你分不清是合理建议还是拿复杂东西糊弄你。每次讨论架构，都让 WorkBuddy 把方案解释清楚：为什么用这个不用那个，好处坏处各是什么，你听明白了、同意了，才让它动手。AI 是执行者，但你必须是那个知道这件事应该怎么做的人。

## 上线之后：运维意识要趁早

产品刚上线时服务器随便扛，用户多起来之后就不一样了：缓存机制、防爬虫、防攻击、流量带宽的费用控制，早期可以不管，到一定量级后不管它就会来找你。公开来源的作者就分享过因未做防护被刷出高额账单的经历。防范意识要趁早，可以让 WorkBuddy 定期帮你检查访问日志和费用异常（见第 26 章可靠自动化）。

## 换成自己的产品时，只改这几处

| 占位内容 | 改成什么 |
|---|---|
| `周记便签`、`物品借用登记` | 你的产品名（先查重再定） |
| `yijie.example.cn`、`203.0.113.10` | 你的真实域名和服务器 IP |
| 一句话需求 | 你的产品核心功能，第一版从简 |
| GitHub | 你的代码托管平台（同样建私有仓库、开分支保护） |
| 云厂商备案入口 | 你购买服务器的平台指引 |

不要删除这些句子：

- 先检查，等我确认后再执行。
- 不要覆盖原文件。
- 无法判断的内容标记"待确认"。
- 每次交付前报告改动的文件清单、功能对照和未解决问题。

## 常见弯路：出问题怎么拉回来

### AI 没等确认就直接开始写代码

```text
请停止，回到规划阶段。先输出 plan.md 让我检查，我回复"确认"之前不要写任何代码。
```

### 新开的对话里 AI 忘了之前的决策，写出了矛盾的代码

```text
请先阅读 README.md 和 docs/decisions.md 再继续。发现你的方案与文档矛盾时，停下来向我指出矛盾点，标记"待确认"，不要自行决定。
```

### 部署失败或网站打不开

```text
请不要反复重试。先只读排查：服务进程状态、端口监听、域名解析、证书有效期、防火墙规则，逐项报告检查结果和可能原因，等我确认后再修复。
```

## 人工确认点

- 规划文档的方向和功能范围，确认后才进入开发。
- 起名和域名查重结论，注册由人完成。
- 备案材料提交、云控制台操作、证书配置。
- 分支保护规则和测试流程是否符合预期。
- 每次部署和合并到主分支之前。

## 安全边界

- 代码仓库一律私有；提交前确认没有写入账号、密钥、身份证等敏感内容。
- 备案、域名注册、服务器购买涉及真实费用和法律责任，由人操作，不让 AI 代付或代提交。
- 涉及真实用户数据时，先做脱敏和访问控制；内部工具不要把公司数据放进未批准的公网服务。
- AI 生成的代码不等于安全代码，对外提供服务前请专业人员做安全检查。
- 各平台入口、功能和规则可能变化，以官方页面为准。本章方法为通用工作方法，不代表公司正式制度。
