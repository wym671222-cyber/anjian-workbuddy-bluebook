# 第 21 章 项目研判分析

项目研判的核心不是让 AI 给出“值得做”或“不值得做”的结论，而是让证据、假设、缺口和风险更清楚。本章使用完全合成的示例介绍通用方法，不展示真实项目或业务体系。

::: warning 使用边界
WorkBuddy 可以辅助整理资料和形成问题清单，但不能替代法律、财务、工程、运营等专业意见，也不能作出投资、审批或项目决策。
:::

## 适用场景

- 对一批项目资料建立目录和完整性检查表。
- 从公开或授权材料中提取关键事实和待核实事项。
- 建立风险清单、假设清单和证据矩阵。
- 为专业评审准备问题清单和汇报结构。

## 任务目标

把“感觉上有问题”转化为可验证的问题，把“材料很多”转化为清楚的证据结构，把结论边界交还给有权限的专业人员。

## 输入材料

- 研判目的、范围、截止时间和使用对象。
- 已授权的项目资料或合成样本。
- 需要关注的维度和专业边界。
- 现有结论、关键假设和缺失材料。
- 期望输出格式与审批要求。

## 执行步骤

### 第一步：建立资料目录

按材料类别记录文件名、日期、来源、版本、完整性和可用状态。损坏、重复、缺页和无法解析的文件单独列出。

### 第二步：形成事实表

每条事实包含内容、来源、时间、可信状态和关联问题。不要把宣传性描述、第三方观点和内部判断混为一类。

### 第三步：建立证据矩阵

围绕研判问题列出支持证据、反向证据、缺失证据和待验证假设。证据不足时，不得强行得出确定结论。

### 第四步：组织风险与问题

风险条目至少说明风险描述、触发条件、可能影响、现有依据、待补材料、建议核验人和状态。AI 只负责整理，不给风险“拍板”。

### 第五步：生成研判材料

输出一页摘要、资料清单、事实表、问题清单和风险清单。任何建议都应说明依据、假设和适用边界。

## 合成示例

假设需要评估“某闲置办公楼改造项目”的资料完整性，所有名称和数字均为虚构。WorkBuddy 可以发现权属材料版本不明、改造预算缺少依据、时间计划存在冲突，并形成待补材料清单；是否推进项目由专业评审和审批程序决定。

### 可复制任务模板

```text
请辅助整理当前目录中的项目资料，不作投资、审批或专业结论。

研判目的：{目的}
材料范围：{目录或文件名}
关注维度：{合规/财务/工程/运营/其他}
输出对象：{经办人/评审人员/会议材料}

请执行：
1. 建立资料目录并标记版本、来源、完整性和异常；
2. 形成事实表，每条事实注明出处；
3. 区分事实、第三方观点、假设和建议；
4. 建立支持证据、反向证据、缺失证据矩阵；
5. 生成风险清单和待核实问题，不自行确定风险等级；
6. 输出一页摘要、材料缺口和建议的专业复核分工。

不得补造项目事实，不得把模型判断写成正式结论，不得修改原始材料。
```

## 人工确认点

- 研判范围、评价口径和资料权限是否明确。
- 关键事实是否有可靠来源，版本是否一致。
- 专业判断是否由对应专业人员完成。
- 风险等级、处置建议和最终结论是否经过授权确认。
- 输出材料是否适合目标读者和共享范围。

## 输出物与验收标准

- 资料目录：文件状态和缺口清楚。
- 事实与证据矩阵：事实有出处，假设有标记。
- 风险清单：描述具体、依据可查、状态可追踪。
- 待核实问题：说明需要什么材料、由什么专业复核。
- 研判摘要：结论边界清楚，不替代正式决策。

## 常见弯路

- **材料没看全就先写结论**：后续容易围绕结论挑证据。先建目录和事实表。
- **只收集支持材料**：忽略反向证据会放大确认偏差。证据矩阵必须包含反证。
- **把专业判断交给模型**：模型不知道完整现场和责任边界。明确专业复核人。

## 安全边界

- 不展示真实项目名称、金额、交易结构、权属信息和业务流程。
- 不把受限资料上传到未经批准的服务。
- 不自动发送报告、修改原始材料或触发业务动作。
- 不把本章方法描述为公司已经正式采用的研判流程。

## 可复用资产

- 项目资料目录模板。
- 事实表与证据矩阵。
- 材料缺口清单。
- 风险与问题台账。
- 研判摘要检查表。
