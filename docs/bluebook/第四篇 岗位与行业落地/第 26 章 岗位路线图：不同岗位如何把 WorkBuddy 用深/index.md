# 第 26 章 岗位路线图：不同岗位如何把 WorkBuddy 用深

岗位应用不应从“这个岗位有哪些提示词”开始，而应从三个问题开始：哪些工作重复发生，哪些产物有稳定标准，哪些决定必须由人承担责任。

## 岗位应用的四级成熟度

| 等级 | 工作方式 | 典型形态 | 进入下一级的条件 |
|-|-|-|-|
| L1 单次辅助 | 人给完整材料，AI 生成草稿 | 纪要、摘要、表格、PPT | 输出可验收，连续成功 3 次 |
| L2 固定模板 | 使用任务卡和标准输出 | 周报、复盘、客户简报 | 输入字段和验收规则稳定 |
| L3 工作流 | Skill、资料库、连接器协作 | 内容流、知识流、报表流 | 有负责人、日志和异常处理 |
| L4 团队能力 | 项目、专家团和自动化运行 | 部门级数字员工 | 权限、成本和资产治理到位 |

每个岗位都应从 L1 的低风险任务开始。不是所有任务都需要进入 L4；低频、差异大或责任重的工作，保持“AI 辅助 + 人工决策”更合理。

## 产品经理

| 场景 | 输入 | 交付物 |
|-|-|-|
| 用户访谈整理 | 录音转写、访谈提纲 | 痛点、原话、待验证问题 |
| 反馈聚类与版本复盘 | 工单、评论、埋点摘要 | 问题分类、影响范围、证据表 |
| 竞品证据库 | 官网、定价、更新日志、评论 | 可追溯对比表和定期简报 |
| 产品发现工作流 | 反馈、数据、竞品、业务目标 | 机会地图和决策材料 |

## 运营与市场

| 场景 | 输入 | 交付物 |
|-|-|-|
| 内容改写、素材盘点 | 已审长文、图片目录 | 平台草稿、素材清单 |
| 标题、封面与平台适配 | 已审事实包、品牌规范 | 标题池、公众号封面、小红书长图、口播稿 |
| 选题与内容中台 | 资讯源、评论、品牌规范、历史内容 | 候选池、证据表、内容日历、复盘库 |
| 内容专家团 | 热点、产品资料、渠道数据 | 研究、脚本、分镜、多平台草稿、复盘 |

## 销售与售前

| 场景 | 交付物 | 风险点 |
|-|-|-|
| 会后纪要 | 需求、异议、待办、负责人 | 把推测写成客户承诺 |
| 客户研究 | 画像、公开动态、机会假设 | 使用过时或同名公司资料 |
| 方案与演示 | 方案大纲、演示脚本、FAQ | 虚构能力、价格和案例 |
| 跟进工作流 | CRM 草稿、提醒、材料更新 | 未确认写入和外发 |

## HR 与行政

| 场景 | 推荐做法 | 必须控制 |
|-|-|-|
| 制度摘要、材料检查 | 基于批准文件生成 FAQ | 版本日期和适用范围 |
| 入职与培训 | 生成学习路径、课件和清单 | 员工数据最小化 |
| HR 知识助手 | 连接制度库回答常见问题 | 权限、引用与升级人工 |
| 人事流程辅助 | 提醒、材料完整性、报表 | 不自动做雇佣决策 |

## 财务与法务

| 财务场景 | 法务场景 | 人工关口 |
|-|-|-|
| 发票识别、账单整理 | 条款提取、版本对照 | 金额与原文复核 |
| 预算差异、异常清单 | 标准条款偏离清单 | 口径和风险判断 |
| 月结材料流水线 | 合同知识库、审阅工作台 | 专业人员签字 |
| 受控系统回写 | 合同流程提醒 | 付款、申报、签署确认 |

## 研发与 IT

| 场景 | 交付物 | 检查点 |
|-|-|-|
| 代码理解、技术文档 | 架构图、README、模块清单 | 与实际代码一致 |
| 测试与缺陷分析 | 测试矩阵、复现步骤、修复建议 | 测试通过、代码评审 |
| 开发工作流 | 分支改动、CI 诊断、发布说明 | 合并和部署确认 |
| IT 自动化 | 巡检、告警摘要、知识库 | 生产权限和事故响应 |

## 部门推广模板

每个岗位先选一个场景，填写下面这张卡：

| 项目 | 填写内容 |
|-|-|
| 原流程 | 谁在什么时间，用哪些系统，花多久 |
| 输入 | 文件、字段、权限、质量问题 |
| 输出 | 文件、系统记录、通知对象 |
| 验收 | 准确性、完整性、时效和风险 |
| 人工关口 | 谁确认，什么情况下必须暂停 |
| 指标 | 基线、目标值、观察周期 |
| 负责人 | 业务、技术、资产维护人 |

岗位落地的终点不是“每个人都会问 AI”，而是形成一批能被新人复用、由负责人维护、出了问题能追溯的标准工作流。
